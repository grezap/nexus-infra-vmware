# deb13 — design + runbook

**Generic Debian 13 base template.** The parent image for roughly 60 of the 65 lab VMs (Vault, Postgres, Kafka, Mongo, Redis, ClickHouse, StarRocks, MinIO, Spark workers, Swarm nodes, …). Each child VM clones this template via `terraform/modules/vm/` and overlays its role-specific Ansible on top.

## Why a base template

Three reasons a shared base beats building each VM from the netinst ISO:

1. **Speed.** Packer build from netinst is ~7 minutes. A `vmrun clone` + NIC rewrite + power-on is ~7 seconds. For 60 VMs, that's the difference between "provision the lab overnight" and "provision the lab in four minutes."
2. **Drift control.** The baseline (chrony client, nftables, node_exporter, sshd hardening, unattended-upgrades, nexusadmin user) is defined in exactly one place — `packer/deb13/ansible/roles/debian_base/`. Fix it once, every clone inherits.
3. **Reproducibility.** The template's `packer-manifest.json` records the sha + timestamp. Every clone is provably from the same source image.

## What the template ships

| Component                      | Status at build time | First-boot behavior |
|--------------------------------|----------------------|---------------------|
| `nexusadmin` sudoer            | Created by preseed; password-auth for Packer | Owner pubkey in `authorized_keys` if `files/nexusadmin.pub` existed at build time |
| `systemd-networkd` on `nic0`   | `.link` rename `OriginalName=en* → nic0`; `.network` DHCP=yes | DHCPs from nexus-gateway's dnsmasq (IP in `192.168.70.200-.250`) |
| `nftables`                     | enabled + running baseline (deny-in, allow SSH + :9100 from VMnet11) | Loads `/etc/nftables.conf` |
| `chrony` (client)              | enabled + running; points at `192.168.70.1` with public pool fallback | Syncs from nexus-gateway once VMnet11 is attached |
| `prometheus-node-exporter`     | enabled + running on `:9100` | Scraped by Phase 0.I Prometheus |
| `unattended-upgrades`          | configured for `Debian-Security` origin only | — |
| Hardened sshd                  | `PermitRootLogin no`, `MaxAuthTries 3`, `ClientAliveInterval 60` | — |
| Owner's SSH key                | Baked in via `files/nexusadmin.pub` | Key-auth works from day one |
| `/etc/machine-id`              | Truncated at build | regenerated by `systemd-machine-id-setup` on first boot |
| SSH host keys                  | Removed at build | regenerated by `ssh-keygen -A` at first `systemctl start ssh` |

## What the template does **not** ship

- No dnsmasq, no DHCP server, no ip_forward, no masquerade — that's `nexus-gateway`'s job.
- No OpenTelemetry Collector — architecture.md defers it to the DRY refactor at Phase 0.B.3 (when a second base-image template arrives and reveals the right shared-role shape).
- No database, no app runtime, no TLS cert tooling — role overlays add those per-VM.

## Build + deploy

Prerequisites:
- Phase 0.B.1 complete (nexus-gateway running at 192.168.70.1 — the clones depend on its DNS/DHCP/NTP).
- `H:/VMS/NexusPlatform/_templates/` directory writable.

```powershell
# 1. Build the template (~7 min)
cd "F:\_CODING_\…\nexus-infra-vmware"
make deb13
# Template lands at H:\VMS\NexusPlatform\_templates\deb13\deb13.vmx

# 2. Smoke-test via the reusable module (~10 sec)
make deb13-smoke
# VM lands at H:\VMS\NexusPlatform\90-smoke\deb13-smoke\deb13-smoke.vmx

# 3. Find its DHCP lease (issued by nexus-gateway's dnsmasq)
#    Assumes handbook §0.4 SSH client setup; otherwise prepend `-i $HOME\.ssh\nexus_gateway_ed25519`.
ssh nexusadmin@192.168.70.1 "awk '\$2==\"00:50:56:3f:00:20\" {print \$3}' /var/lib/misc/dnsmasq.leases"

# 4. Probe it directly (replace <ip> with the lease from step 3)
Test-NetConnection <ip> -Port 22
Test-NetConnection <ip> -Port 9100
ssh nexusadmin@<ip>     # Linux remote shell defaults to bash -- no wrapper needed

# 5. Tear down
make deb13-smoke-destroy
```

## Verification checklist

Inside the smoke-test VM:

```bash
# NIC renamed correctly
ip -br link           # expect `nic0` UP with DHCP-assigned IP

# Time synced via the gateway
chronyc sources -v    # 192.168.70.1 should show as the preferred source

# nftables loaded
sudo nft list ruleset | head -40

# node_exporter responds on :9100
curl -s localhost:9100/metrics | head -5

# DNS works (forwarded by nexus-gateway's dnsmasq)
nslookup debian.org
```

## Design decisions worth remembering

### NIC rename uses `OriginalName=en*`, not MAC match

Unlike `nexus-gateway` (which pins MACs in the `.link` file because it has three NICs that must map to specific subnets), the `debian_base` role uses `[Match] OriginalName=en*`. This makes the `.link` file **MAC-agnostic** — the same template serves any caller regardless of which MAC they assign. Every clone's single NIC renames to `nic0` deterministically.

The alternative — baking a specific MAC into the template's `.link` file — would mean only the first-instantiated clone's NIC renames; every subsequent clone (with a different MAC) would keep `ensXXX` and fail to bind the `.network` file. Would have forced Terraform to rewrite `.link` files per clone. `OriginalName=en*` is cheaper and equally stable.

### Chrony has a public-pool fallback

`files/chrony.conf` lists `server 192.168.70.1 iburst prefer` **and** `pool 2.debian.pool.ntp.org iburst`. The `prefer` keyword biases toward the gateway once available, but the public pool is the fallback for:
- Packer build time (the NIC is on NAT, not VMnet11 → `192.168.70.1` unreachable).
- Disaster recovery when nexus-gateway itself is being rebuilt.
- Cold-boot races where chrony starts before the DHCP lease is up.

Without the fallback, chrony would churn on "no usable sources" and `makestep` never fires — the VM's clock drifts until you notice.

### SSH host keys are deliberately removed at build time — and how we regenerate them

The Packer final-cleanup step does `sudo rm -f /etc/ssh/ssh_host_*`. If we didn't do this, every clone would share host keys → `ssh` would warn "POSSIBLE DNS SPOOFING" when you moved between clones, and nothing involving SSH trust-on-first-use would work sanely.

**The catch:** Debian 13's stock `ssh.service` has `ExecStartPre=/usr/sbin/sshd -t`, which fails (without the `-` dash prefix, so fatally) if no host keys exist. A naïve additional `ExecStartPre=-/usr/bin/ssh-keygen -A` in a drop-in doesn't help — systemd **appends** drop-in `ExecStartPre` lines rather than prepending, so `sshd -t` runs first, fails, and our regen never fires.

**The fix** (`debian_base` role — `/etc/systemd/system/ssh.service.d/10-regenerate-host-keys.conf`):

```ini
[Service]
# Clear inherited ExecStartPre list, then re-declare in order:
ExecStartPre=
ExecStartPre=-/usr/bin/ssh-keygen -A
ExecStartPre=/usr/sbin/sshd -t
```

A bare `ExecStartPre=` resets the list; we then re-add our regen first and the validation test second. The `-` dash on `ssh-keygen -A` makes it tolerant of already-existing keys (idempotent — boots after the first are a no-op). Verified on a clean clone: `systemctl status ssh` reports both pre-steps as `status=0/SUCCESS`, `Active: active (running)`, `Server listening on 0.0.0.0 port 22`.

### No sshd_config replacement — use `sshd_config.d/` instead

The role drops a `/etc/ssh/sshd_config.d/10-nexus-hardening.conf` snippet rather than rewriting `/etc/ssh/sshd_config`. Debian 13's stock `sshd_config` ends with `Include /etc/ssh/sshd_config.d/*.conf`, so this just works. Advantage: package upgrades that touch `sshd_config` don't fight our edits.

## Rebuild procedure

```powershell
cd "F:\_CODING_\…\nexus-infra-vmware"
make deb13-smoke-destroy                                    # if a smoke VM exists
Remove-Item -Recurse -Force H:\VMS\NexusPlatform\_templates\deb13 -ErrorAction SilentlyContinue
make deb13                                                  # rebuild template
make deb13-smoke                                            # verify
```
